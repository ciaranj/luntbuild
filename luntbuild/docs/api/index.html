Chapter 2. Atom Feed

Atom 1.0 is a type of web feed format used to syndicate lists or collections of information. In Luntbuild, you can obtain notification messages for schedules and builds in Atom 1.0 format.

To use the feed, point a feed reader (almost all modern browsers include a reader) to the following URL or one of the other URLs described in this chapter. 

http://<hostname>:8080/luntbuild/api/atom

Notify Strategy

By default, the web feeds for build notifications will follow the notify strategy set for each schedule. Only builds matching the notify strategy will be included in the feed. This behavior can be overridden by setting a notify strategy for the feed using the notify parameter (aliases can be used instead of the full text): 

http://<hostname>:8080/luntbuild/api/atom/builds?notify=<strategy>

Notify Strategies:

StrategyAliasDescription
notify alwaysalwaysIncludes all builds.
notify when successsuccessIncludes all builds that succeeded.
notify when failedfailedIncludes all builds that failed.
notify when status changedchangedIncludes all builds that changed status from the last build.
notify when failed or status changedfailedorchangedIncludes all builds that failed or changed status from the last build.

Schedules

http://<hostname>:8080/luntbuild/api/atom/schedules/<source>

This feed will include notifications for the specified schedules. Possible values for <source> are:

  • blank - show all builds on this server

  • project - show only schedules from this project

No other options will work.

Builds

http://<hostname>:8080/luntbuild/api/atom/builds/<source>

This feed will include notifications for the specified schedules. Possible values for <source> are:

  • blank - show all builds on this server

  • project - show only schedules from this project

  • project/schedule - show only builds from this schedule

No other options will work.

Build

http://<hostname>:8080/luntbuild/api/atom/build/<project>/<schedule>/<version>

This feed will only include the notification for the specified build. The project name, schedule name, and build version are all required.

Users

http://<hostname>:8080/luntbuild/api/atom/users

This feed will include all users on the server.

User

http://<hostname>:8080/luntbuild/api/atom/user/<user>

This feed will only include the user specified. No other options will work.

Chapter 3. JSON Object

JavaScript Object Notation (JSON) is a type of data-interchange language:

JSON (JavaScript Object Notation) is a lightweight data-interchange format. It is easy for humans to read and write. It is easy for machines to parse and generate. It is based on a subset of the JavaScript Programming Language, Standard ECMA-262 3rd Edition - December 1999. JSON is a text format that is completely language independent but uses conventions that are familiar to programmers of the C-family of languages, including C, C++, C#, Java, JavaScript, Perl, Python, and many others. These properties make JSON an ideal data-interchange language.

--http://www.json.org/

You can obtain almost all information stored in Luntbuild in JSON format. JSON format is particularly useful in JavaScript because it is automatically recognized and parsed. Once parsed, the information will be available as any normal object structure with basic data types.

To see the object structure generated by Luntbuild, see the JSON object page. This structure is almost exactly the same as the XML output. 

http://<hostname>:8080/luntbuild/api/json

The default request includes all information available (system information, all projects, schedules and builds). The results in the root object can be limited by using certain functions described below. Any errors that are encountered will be included in the error member of the root object.

Callbacks

To ease use with JavaScript, any JSON response can be wraped in a call back function by providing a value for the callback paremeter in the request.

Here some JavaScript that shows the basics of using JSON: 

<script type="text/javascript">
function test(luntbuild)
{
   alert(luntbuild.system.servletUrl);
}
</script>
<script type="text/javascript" src="http://<hostname>:8080/luntbuild/api/json?callback=test"></script>

The JSON response is included as an external script. Luntbuild will return the responce wraped in a call to the function test. The JavaScript parser will automatically parse the JSON text and provide the root object to the fuction.

Projects

http://<hostname>:8080/luntbuild/api/json/projects

This request will include a list of projects on the server, but not their schedules or builds.

Schedules

http://<hostname>:8080/luntbuild/api/json/schedules/<source>

This request will include a list of schedules according to <source>, but not their builds. Possible values for <source> are:

  • blank - show all schedules on this server

  • project - show only schedules from this project

No other options will work.

Builds

http://<hostname>:8080/luntbuild/api/json/builds/<source>

This request will include a list of builds according to <source>. Possible values for <source> are:

  • blank - show all builds on this server

  • project - show only builds from this project

  • project/schedule - show only builds from this schedule

No other options will work.

Build

http://<hostname>:8080/luntbuild/api/json/build/<project>/<schedule>/<version>

This request will only include the specified build as the result. The project name, schedule name, and build version are all required.

Users

http://<hostname>:8080/luntbuild/api/json/users

This request will include a list of all users on the server in the results.

User

http://<hostname>:8080/luntbuild/api/json/user/<user>

This request will only include the user specified in the results. No other options will work.

Chapter 6. Using Luntbuild External API

Table of Contents

JavaDoc

Luntbuild provides a suite of remoting API with the following abilities:

  1. Create or edit a project.

  2. Trigger a build in any build schedule.

  3. Configure properties of projects, as well as global properties.

  4. Search builds and get a build information such as its artifacts url, etc.

By utilizing the Hessian web service protocol, this API is very easy to use. Basically two jar files needs to be included in the classpath, hessian-3.0.8.jar and luntbuild-api.jar. They can be found in the "remoting" directory. There are also some examples to demonstrate usage of the API. The remoting API Javadoc is available here. Following examples are available:

  1. Edit Properties - This class gives an example of using luntbuild web service API to edit propertie of a project

  2. Search Builds - This class demonstrates search builds in the system, and how to access build information

  3. Trigger Build - This class gives an example of using luntbuild web service API to trigger a build

  4. LuntbuildConnection- This class demonstrates access to project, schedule and build data. It shows how to create, edit and delete a project (supported since Luntbuild 1.3). This class is part of the Luntclipse, Eclipse plugin for Luntbuild.

The provided TriggerBuild example can be used to implement a real-time Continuous Integration, that is, whenever a checkin is made into the repository, Luntbuild can trigger immediate build. We create a sample Cvs repository to show how to do this:

  1. Create a manually triggered schedule in a project you will use to implement real-time continuous integration. To get the build fast, you can configure the build to build incrementally.

  2. Checkout "loginfo" file under CVSROOT directory of your cvs repository, and append a line like this: 

    testcvs cmd /c d:/lunt/cvs/lunt/luntbuild/remoting/samples/trigger_build.bat

    Note

    Before editing, this file should be checked out first using your cvs client, just like you edit other files in your cvs repository.

    where testcvs should be replaced with the directory path of your Cvs repository. All checkins under this path will trigger the trigger_build.bat command. The file trigger_build.bat resides in the "remoting/samples" directory. Of course, you can copy the related files (the remoting API jar files, the TriggerBuild.class file, and the trigger_build.bat file) to any other machine as long as JDK1.4 or higher are installed. The path to trigger_build.bat need to be changed to fit your environment. Contents of trigger_build.bat should be changed to reflect proper classpaths, your Luntbuild server url, your desired project, and build schedule. On Unix platform, you can create the trigger_build.sh script easily based on the contents of the file trigger_build.bat.

  3. Check in the "loginfo" file. From now on, the checkins under your configured path will trigger the trigger_build command, which will result in a build you have configured above.

Note

Any other Version Control System that can trigger an external command on checkin can use this method to implement real-time Continuous Integration.
